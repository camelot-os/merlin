# Merlin driver framework for Camelot user drivers

<img src="doc/concepts/_static/figures/merlin.png" width="200">

## Basics

Merlin is a driver framework to unify as much as possible the common actions and tooling requires for all drivers.

Merlin is responsible for typical actions:

- forge the task declared devices list based on the input device tree
- deliver various helpful getters on various DTS or SVD related fields, in a generic way for any SoC/Arch
- allow easy driver registering and usage, making the driver developer to implement only the HW IP specific code

## Merlin architectural model

All in-SoC devices are regrouped using the plaform interface of Merlin, declared in the `<merlin/platform/driver.h>`
header. Various in-S devices such as buses are based on this interface for their declaration

## Examples

Merlin repository hold some examples drivers to show how to use the Merlin driver framework is concrete cases.

### I2C drivers examples

Merlin delivers two typical drivers implementations based on Merlin driver framework API

   * A typical STM32 I2C bus controller driver for I2C v2 controller such as in U5A5 STM32 subfamily
   * A typical I2C slave driver using the above i2c bus API, for ILI2130 Touch screen

## Rust crate: production and Meson integration

The repository can build the Rust crate `camelot-merlin` from the project root
through Meson.

### Scope

When enabled, Meson drives Cargo to build the crate for embedded target
`thumbv8m.main-none-eabi` and wires the project DTS inputs into the Rust build
script.

The Cargo package itself lives in `rust/` and can still be built standalone.

### Prerequisites

- Meson >= 1.7.0
- Python 3 (used by Meson custom target wrapper)
- Cargo and Rust toolchain available in PATH
- Rust target installed:

```bash
rustup target add thumbv8m.main-none-eabi
```

### Meson options used by Rust integration

- `-Dwith_rust=true`: enables crate build from Meson
- `-Ddts=<file>`: selects the DTS input used by devicetree flow
- `-Ddts-include-dirs=[...]`: include search roots forwarded to Rust DTS generation

The Rust integration is disabled by default (`with_rust=false`).

### What Meson does when `with_rust=true`

At configure/build time, Meson creates a `rust-crate` custom target with the
following behavior:

1. Uses Cargo in `rust/`
2. Builds for `thumbv8m.main-none-eabi`
3. Generates a stamp output `rust-crate.stamp` in build directory
4. Rebuilds when the Meson DTS input changes

Meson also exports these environment variables to Cargo/build.rs:

- `MERLIN_DTS`: full path to selected DTS input
- `MERLIN_DTS2SRC`: full path to `dts2src` binary from devicetree subproject
- `MERLIN_DTS_INCLUDE_DIRS`: `:`-joined include roots from `-Ddts-include-dirs`
- `MERLIN_CONFIG_TASK_LABEL`: currently forced to `0x0` by Meson root build

### End-to-end Meson example

```bash
meson setup builddir \
   -Dwith_rust=true \
   -Ddts=dts/sample.dts \
   -Ddts-include-dirs=['subprojects/devicetree/include','subprojects/devicetree/dts/common']

ninja -C builddir
```

The Rust custom target is `build_by_default`, so a normal `ninja` includes crate
production automatically.

### Building the crate standalone (without Meson)

From `rust/`, a standalone Cargo flow is still supported.

Minimal host-side checks:

```bash
cd rust
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
RUSTDOCFLAGS='-D warnings' cargo doc --no-deps --all-features
cargo test --doc --all-features
```

Cross build with DTS generated by Meson (recommended):

```bash
cd rust
MERLIN_DTS=../builddir/subprojects/devicetree/sample.dts.pp \
MERLIN_CONFIG_TASK_LABEL=0xC001F002 \
cargo clippy --target thumbv8m.main-none-eabi -- -D warnings
```

### DTS generation behavior in Rust build script

`rust/build.rs` generates `dts_generated.rs` from template and DTS metadata.

- If `MERLIN_DTS` is set: generation uses `dts2src`
- If DTS parsing fails first pass: build script retries with C preprocessor (`cpp`)
- If `MERLIN_DTS` is not set: an empty fallback backend is generated so the crate
   still compiles

For crate-level details (API, feature flags, standalone usage), see `rust/README.md`.
